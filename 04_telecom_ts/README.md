# TelecomTS → CPT World-Knowledge JSONL

> **Status: In progress** — demo output generated; blocked on team review (GPT-generated anomaly text, lab vs. production data). Do not scale until approved.

5G lab testbed KPI time series paired with network scenario descriptions and anomaly troubleshooting text for continued pre-training (CPT).

---

## What this dataset is

[TelecomTS](https://huggingface.co/datasets/AliMaatouk/TelecomTS) is a public 5G observability dataset collected from a **controlled lab testbed** — not a live production network. Each sample is a fixed **128-timestep window** (12.8 seconds at 10 Hz / 100 ms sampling) across **18 KPI channels**: RSRP, BLER, MCS, PRB utilization, byte counts, packet counts, SNR, buffer estimates, and protocol indicators.

Each sample includes a `description` (prose summary of the KPI window written by dataset authors), anomaly metadata with troubleshooting tickets (for anomalous samples), and scenario labels (zone, traffic type, mobility, congestion).

---

## ⚠️ Things to discuss before scaling this up

These are the issues to raise with Charon:

**1. This is lab data, not real production telemetry.**
The experiment was run in a university lab with a single base station and a handful of phones. It's not carrier-grade data from a live network. The patterns may not transfer to real NOC scenarios.

**2. Most anomalies are fake — they were mathematically injected.**
Researchers took normal measurements and added artificial signal patterns to simulate 10 types of failures (antenna failure, jamming, congestion, etc.). Only the jamming events used a real physical jammer. Everything else is simulated by modifying the numbers. So the model would learn about synthetic, controlled failures rather than organic real-world ones.

**3. The anomaly troubleshooting text was written by GPT-4.1, not engineers.**
The text explaining what went wrong in anomaly samples was generated by GPT-4.1, not by a real network operator. We tag all anomaly records with `text_quality: "generated"` and `text_source: "generated_gpt4"` to be transparent about this. The `description` text (which describes the normal KPI pattern) is real — written by the dataset authors — and is tagged `text_quality: "real"`.

**4. The "description" text reads like a stats report, not natural operator prose.**
It says things like "RSRP rose by +1.2 dB/s over indices 1–53." This is computed from the signal, not written by a human observing the network. It's accurate but somewhat mechanical.

**5. Template suffix is appended to every record.**
The script adds a fixed closing sentence before `<ts></ts>` describing the channel list. This is the same across all records of the same type. The description part is unique per sample, but the closing line is templated.

**Bottom line:** This dataset is fine as a diversity-add for observability/telecom domain coverage, but flagged before large-scale CPT use. The synthetic anomalies + GPT text for the most interesting portion (failures) is the main concern.

## What we are training for

**Objective:** Continued pre-training on world knowledge, not instruction fine-tuning. The model reads natural text with a `<ts></ts>` placeholder; five KPI series are stored separately. There is no forecast prompt.

**What the model learns:** How network observability prose relates to KPI patterns — e.g. that rising BLER with falling MCS during a jamming event corresponds to specific throughput and signal-strength traces; how file-download vs YouTube traffic shapes byte-count asymmetry.

**Two record types** (separated by `record_type`):

| Type | Text source | When |
|------|-------------|------|
| `normal` | `description` only | No anomaly in sample |
| `anomaly` | `description` + troubleshooting ticket | `anomalies.exists == true` |

## Window design

**One sample = one record.** Each Hugging Face row is already a fixed 128-timestep window. No sliding windows.

- **128 timesteps × 100 ms = 12.8 seconds** of KPI data per record
- Normal and anomaly samples are exported as separate record types
- Default demo run: `max_records: 50` with `balance_types: true` → 25 normal + 25 anomaly

---

## The five time series

Fixed order (one value per 100 ms timestep, 128 values each):

| Order | Source KPI | Output unit | Why included |
|:---:|---|---|---|
| 1 | `RX_Bytes` | `DL_Throughput_Mbps` | Downlink traffic volume → throughput proxy |
| 2 | `TX_Bytes` | `UL_Throughput_Mbps` | Uplink traffic volume → throughput proxy |
| 3 | `DL_BLER` | `DL_BLER_pct` | Downlink block error rate — link quality |
| 4 | `UL_BLER` | `UL_BLER_pct` | Uplink block error rate — often first to degrade |
| 5 | `RSRP` | `RSRP_dBm` | Reference signal strength — coverage indicator |

**Not included:** PRB utilization, MCS, SNR, and packet counts — available in the raw dataset but omitted to keep a focused 5-channel panel. Throughput is derived from byte counts (`bytes × 8e-5` → Mbps per 100 ms sample). BLER values are scaled to percent (`× 100`).

## Quick start

```bash
cd datasets/04_telecom_ts
pip install -r requirements.txt

# Demo export: 25 normal + 25 anomaly (50 total)
python scripts/build_cpt_jsonl.py --config config.example.yaml

# Dry-run: print one example record, no files written
python scripts/build_cpt_jsonl.py --dry-run

# Anomaly types only
python scripts/build_cpt_jsonl.py \
  --set data.record_types=[anomaly] \
  --set 'data.anomaly_types_filter=[Jamming,Antenna Failure]' \
  --set output.max_records=20

# Custom channel selection
python scripts/build_cpt_jsonl.py \
  --set data.channels=[RSRP,DL_BLER,UL_BLER,UL_SNR,RX_Bytes]
```

Outputs:

- `output/telecom_ts_cpt.jsonl` — CPT records
- `output/run_report.json` — counts, anomaly breakdown, config snapshot

## Configuration guide

| Section | Key | Purpose |
|---------|-----|---------|
| **data** | `hf_dataset_id` | Hugging Face dataset ID |
| | `cache_dir` | Local HF cache path |
| | `split` | Dataset split to stream |
| | `record_types` | `normal`, `anomaly`, or both |
| | `channels` | KPI column names (5 default) |
| | `anomaly_types_filter` | Restrict anomaly types; `[]` = all |
| | `window_length` | Expected timesteps (128) |
| | `freq` | Timeseries frequency label (`100ms`) |
| **output** | `output_path` / `report_path` | Output files |
| | `max_records` | Cap total records; `null` = no cap |
| | `balance_types` | Split cap evenly across record types |
| | `indent` | `null` = compact JSONL |
